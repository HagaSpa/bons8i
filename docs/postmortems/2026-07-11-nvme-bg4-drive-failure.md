# Postmortem: NVMe 障害の再発と root cause の改訂（APST 説の棄却 → BG4 ドライブ障害と確定）

- **日付**: 2026-07-11（再発の発見・切り分け・解決を同日で完了）
- **環境**: Raspberry Pi 5 8GB / Ubuntu 24.04 arm64 / kubeadm シングルノードクラスタ / Waveshare PoE M.2 HAT+ (B)
- **先行文書**: [2026-07-10-nvme-apst-hang.md](./2026-07-10-nvme-apst-hang.md)（前日に「APST 相性・解決済み」と結論した初報。**本文書はその root cause を覆す続報**。初報は当時の判断材料の記録として改変せず残す）
- **ステータス**: 解決済み（**ドライブ交換 + NVMe データディスク化まで完了**）

## 要約

前日「APST（自律的省電力遷移）と Pi 5 の相性問題」と結論しカーネルパラメータで対処した NVMe 消失障害が、対処の約20分後に再発していたことが翌朝判明した。切り分けの結果、ソフトウェア説をすべて棄却し、**KIOXIA BG4 ドライブ自体の障害**と確定。手持ちの Intel 660p 512GB に交換し、データディスク化・既存 PV 移行・再起動試験まで完了した。

## 調査

### 死亡 #2 の法医学的証拠（再発発見の決め手）

dmesg のエラー位置がすべてを語っていた:

```
[ 1233.720466] nvme nvme0: controller is down; will reset: CSTS=0xffffffff, PCI_STATUS=0xffff
[ 1237.006644] nvme0n1: I/O Cmd(0x2) @ LBA 126976, 2048 blocks, I/O Error
```

- 起動から **1233 秒（約20分）**で死亡 = 「長期アイドルで深い眠り」というAPST 説の前提と矛盾
- 失敗した読み出しは **LBA 126976 ≒ 65MB 地点・1MB 単位** = 実行中だった dd（`bs=1M`）の進行位置と一致。`/proc/diskstats` の凍結値（~72MB）とも整合 → **負荷中の死**と確定

### 「37秒」の正体（死亡 #3〜#5 の共通パターン）

再現試験の dd はすべて**約37秒**で I/O エラーを返したが、読めた量は 10/134/56MB とバラバラだった。37秒 = NVMe ドライバの **I/O タイムアウト（30秒）+ リセット試行**の時間であり、実際は**負荷開始から数秒でコントローラが応答停止**していた。また速度が健全時の1%未満（1.6〜3.8MB/s）で、**最初からリトライまみれ**だった = コントローラの末期的挙動。

### 切り分けマトリクス（変数を1つずつ）

| テスト | 結果 | 除外できた仮説 |
|---|---|---|
| カーネルパラメータ3点適用 | 死亡 #3 | APST / ASPM / ポート PM（ソフトウェア説すべて） |
| SSD 再装着 | 死亡 #4 | M.2 スロットの接触不良 |
| FFC ケーブル両端再装着 | 死亡 #5 | FFC の接触不良 |
| AER（PCIe エラー記録）確認 | 記録なし | （判別材料にならず） |
| dmesg の undervoltage 確認 | 記録なし | （電源説は弱い証拠のみで残存） |
| **A/B 換装（Intel 660p）** | **1GB 完走・286MB/s・再列挙正常** | **Pi 5 / HAT / FFC / 電源のすべて** |

同一プラットフォームでドライブだけ替えたら全快 = **問題は BG4 側**と確定。

### HAT の ACT LED の意味（最終確定）

前日の同定に加え、660p の正常動作の観察で完成:

| HAT 緑 LED | 意味 |
|---|---|
| 消灯 | 生存・アイドル |
| I/O に連動して点滅 | 正常な読み書き中 |
| **ホスト I/O なしで点滅継続** | **コントローラのハング/死亡** |

前日「ファームウェアの自律動作」と解釈した点滅は、**死亡状態の表示**だった可能性が高い（死亡 #2 の直後の観察だったため）。

## 根本原因（改訂版）

**KIOXIA BG4 ドライブの障害。** コントローラが連続 I/O 数秒〜、アイドルでも数日で応答停止する。「個体不良」か「BG4 × Pi 5 の相性」かは未分離（分離には別マシンでのテストが必要。コミュニティ情報では BG4 は Pi 5 での動作実績が薄く、不安定・高温の報告あり）。どちらであっても**このプラットフォームで BG4 は使用不可**という運用判断は変わらない。

## 対処（最終）

1. **ドライブ交換**: KIOXIA BG4 256GB → **Intel 660p 512GB**（手持ち品を転用。SMART: Percentage Used 14%・Media Errors 0。消去後に 4GB 直書き 331MB/s 完走で合格判定 = **今回は負荷試験に合格してから信を置いた**）
2. **データディスク化**: ext4（ラベル `k8s-data`）を **local-path-provisioner の既定パス `/opt/local-path-provisioner` に直接マウント** → K8s 側の設定変更ゼロで保存先が microSD → NVMe に切替
3. **既存 PV データの移行**: 書き手（Forgejo StatefulSet）を `scale --replicas=0` → `cp -a` で NVMe へ → マウント切替 → `scale --replicas=1` → UI・リポジトリの無事を確認
4. **fstab に `nofail,x-systemd.device-timeout=10`**: 万一ドライブが再び死んでも**起動がマウント待ちでハングしない**保険（本障害の直接の教訓）
5. **再起動試験**: 自動マウント + Forgejo 自動復帰を無人で通過

カーネルパラメータ3点は不要になったが害が小さいため当面残置（掃除は後日）。

## 教訓

- **「解決した」の判定には負荷試験を含める**: 認識が戻った・SMART が PASSED、は解決の証明ではない。初報はここで踏み外した。今回の 660p は書き込み試験合格後に採用した
- **対処直後の復活を対処の効果と混同しない**: 電源サイクル自体が症状を一時解消する障害では、対処の有効性は再現試験でしか確認できない
- **もっともらしい示唆（カーネルメッセージ含む）も仮説にすぎない**: エラーメッセージの「Try ...」は診断ではなくヒント。示唆に沿った対処が効かないことで初めて仮説が棄却できた
- **切り分けは変数1つずつ + 対照実験（A/B）が最強**: 5回の死亡はすべて診断情報になった
- **初報を上書きせず続報を出す**: 「当時の判断材料で何が合理的だったか」を残すことで、誤診のメカニズム自体が学習資産になる（本文書の形式）
- **検知の不在が今回も根本問題**: 死亡 #2 に半日気づかなかった。監視スタック（VictoriaMetrics + Alertmanager → GitHub issue 自動起票）導入の優先度を再確認
